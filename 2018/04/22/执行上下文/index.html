执行上下文

发表于 2018-04-22 | 分类于 JavaScript ， 函数

执行上下文，指的是代码的运行环境。一段代码什么时候运行，运行结果是什么，取决于代码当前所处的运行环境，即当前的执行上下文。

分类

在 JavaScript 中代码的运行环境分为三种：

  • 全局上下文：代码的默认运行环境。一旦代码被载入，就会立即产生一个全局上下文。
  • 局部上下文：当一个函数被调用时，就会产生一个函数体内的上下文，叫做局部上下文。
  • eval 上下文：eval函数被调用时内部产生的上下文（不考虑）。

执行上下文栈

程序中产生的所有执行上下文都会保存在内存栈空间中。但是，如果一个程序一直产生很多的执行上下文，那内存空间万一不够了怎么办？

压栈出栈

执行上下文在栈空间中的存储其实是有一个“压栈出栈”的过程的。

“压栈”即当执行上下文产生后会存入栈空间中；“出栈”即当执行上下文中的代码运行完成后，该执行上下文会从栈空间中销毁。

所以，栈空间中的执行上下文其实一直在不断的产生和销毁，所以一般情况下并不造成内存空间不足的情况。

压栈出栈过程

因为 JavaScript 是单线程语言。单线程放在现实生活就表现为同一时间内只能做一件事。那么，在 JavaScript 中，单线程就体现在，同一时间内栈空间中处于活动状态的执行上下文只能有一个。

但是，在一段程序中，产生的执行上下文肯定不止一个，那么怎么才能让栈内存中处于活动状态的执行上下文始终只有一个呢？

栈空间的存储原理是先进后出。执行上下文压栈是从下往上压栈，即最先进入栈空间的执行上下文会被保存在最底部，最后进入栈空间的执行上下文会被保存在最顶部。而执行上下文出栈是从上往下出栈，即只能从最顶层开始往下销毁。因此，对于执行上下文来说，处于活动状态的永远都只能是最顶层的执行上下文。

我们通过一段代码来具体看一下执行上下文的压栈出栈过程。

第一步：全局上下文压栈。

全局上下文压栈后，全局上下文进入活动状态，其内部可执行代码开始执行。执行到第 7 行代码outer()时，函数outer被调用，激活函数outer创建它自己的执行上下文。

第二步：函数outer的执行上下文压栈。

outer的上下文压栈后，outer的上下文进入活动状态，outer函数内部的可执行代码开始执行。执行到第 5 行代码inner()时，函数inner被调用，激活函数inner创建它自己的执行上下文。

第三步：函数inner的执行上下文压栈。

inner的上下文压栈后，inner的上下文进入活动状态，inner函数内部的可执行代码开始执行。然后没有再遇到其他能生成执行上下文的情况，因此函数inner内的代码顺利执行完毕，inner的上下文销毁出栈。

第四步：函数inner的执行上下文出栈。

inner的上下文销毁之后，函数outer的执行上下文重新回到活动状态，继续执行其内部可执行代码，然后也没有再遇到其他能生成执行上下文的情况，因此函数outer内的代码也顺利执行完毕，outer上下文销毁出栈。

第五步：函数outer的执行上下文出栈。

outer的上下文销毁之后，全局上下文重新回到活动状态，继续执行其内部可执行代码，然后也没有再遇到其他能生成执行上下文的情况。直到浏览器关闭，全局上下文销毁出栈。

注意：在函数中，如果遇到return会直接终止函数内可执行代码的执行，因此函数调用也会立即完成，所以函数的上下文也会立即从栈空间中销毁。

总结

了解完整个过程之后，我们就可以对执行上下文总结一些结论了。

  • 单线程。
  • 同步执行，处于活动状态的永远都只能是最顶层的执行上下文，只要栈顶的上下文处于执行中，其他上下文需要等待。
  • 全局上下文只有唯一的一个，在浏览器打开时创建入栈，在浏览器关闭时销毁出栈。
  • 局部上下文个数没有限制，在每次函数被调用时创建入栈，函数执行完成后销毁出栈。
  • 同一个函数被调用多少次，就会产生多少执行上下文，即使是自己调用自己。
